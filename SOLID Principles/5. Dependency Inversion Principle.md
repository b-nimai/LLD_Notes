# Dependency Inversion Principle (DIP) — the **D** in SOLID

> **One-liner:** **High-level and low-level modules should both depend on abstractions**, not on each other — and abstractions shouldn't depend on details, details should depend on abstractions.

**Trigger phrase:** "Don't `new` a concrete class inside your logic — depend on an interface and have it injected. Code to the contract, not the implementation."

---

## Prerequisites (vocabulary)

- **High-level module** — the core logic / "the brains"; makes decisions and coordinates features. *Analogy: the CEO.*
- **Low-level module** — the details: DB access, API calls, file I/O, data providers. *Analogy: the employees.*
- **Abstraction** — an interface/abstract class both sides agree on (the contract).

**The "inversion":** normally high-level code directly depends on (and creates) low-level code. DIP **flips** that — both now point at an abstraction in the middle, so the dependency arrow no longer runs top→down into details.

---

## Core Idea

Rather than high-level classes controlling and depending on the details of lower-level ones, **both depend on an interface**. This makes code **flexible, testable, and maintainable** — you can swap, mock, or upgrade the low-level part without touching the high-level logic.

---

## Real-life Analogy — Food Delivery

Hungry for pizza? You use a **food delivery app**, you don't call the chef. You don't care *which* chef cooks it or *who* delivers — you depend only on the **app (abstraction)**.

```
You (high-level) → depend on → Food App (abstraction) ← implemented by ← Restaurant (low-level)
```

## Worked Example — Netflix Recommendation Engine

Strategies: Recently Added · Trending Now · Genre-Based (more coming).

### ❌ Without DIP — tightly coupled
```java
class RecentlyAdded {
    public void getRecommendations() { System.out.println("Showing recently added content..."); }
}

class RecommendationEngine {
    private RecentlyAdded recommender = new RecentlyAdded();   // hard-wired concrete dependency
    public void recommend() { recommender.getRecommendations(); }
}
```
**Problem:** `RecommendationEngine` is welded to `RecentlyAdded`. Switching to `TrendingNow`/`GenreBased` means **editing the engine** (and it also breaks [[2. Open Closed Principle]]).

### ✅ With DIP — depend on an abstraction
```java
interface RecommendationStrategy {                 // the abstraction both sides depend on
    void getRecommendations();
}

class RecentlyAdded implements RecommendationStrategy {
    public void getRecommendations() { System.out.println("Showing recently added content..."); }
}
class TrendingNow implements RecommendationStrategy {
    public void getRecommendations() { System.out.println("Showing trending content..."); }
}
class GenreBased implements RecommendationStrategy {
    public void getRecommendations() { System.out.println("Showing content based on your favorite genres..."); }
}

class RecommendationEngine {                        // high-level module
    private RecommendationStrategy strategy;
    public RecommendationEngine(RecommendationStrategy strategy) {  // injected, not new-ed
        this.strategy = strategy;
    }
    public void recommend() { strategy.getRecommendations(); }
}

// main: choose & inject the strategy from outside
RecommendationStrategy strategy = new TrendingNow();   // or RecentlyAdded / GenreBased
new RecommendationEngine(strategy).recommend();
```
The engine **doesn't know or care** how recommendations are produced; strategies are swapped or upgraded freely.

### Swapping at runtime
```java
RecommendationEngine engine = new RecommendationEngine(new GenreBased());
engine.recommend();   // just pass a different strategy — engine untouched
```

## DIP vs Dependency Injection (don't confuse them)

- **DIP** = the **principle**: depend on abstractions.
- **Dependency Injection (DI)** = the common **technique** to achieve it — hand the dependency in from outside (constructor/setter) instead of `new`-ing it inside.

In the example, passing the `RecommendationStrategy` into the constructor **is** DI, serving DIP.

---

## Benefits

- **Flexibility** — swap implementations without touching high-level code.
- **Testability** — inject **mocks/stubs** for the abstraction in unit tests.
- **Reusability** — logic isn't bound to one concrete implementation.
- **Maintainability** — change one part without disturbing others.
- **Scalability** — upgrade/scale pieces without a rewrite.

---

## Pitfalls / Things to Remember

- **The smell is `new SomeConcreteClass()` inside business logic.** Move creation to the edges (a factory / DI container / `main`).
- DIP needs a **stable abstraction**; inverting onto an interface that keeps changing buys little.
- DIP ties the bow on SOLID: it's the mechanism behind [[2. Open Closed Principle]] and pairs with [[4. Interface Segregation Principle]] (depend on *small* abstractions).
- Don't invert *everything* — abstract the dependencies that actually vary or need mocking (respect YAGNI).
