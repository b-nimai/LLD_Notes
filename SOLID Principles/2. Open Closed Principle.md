# Open/Closed Principle (OCP) — the **O** in SOLID

> **One-liner:** Software entities should be **open for extension, but closed for modification** — add new behavior by adding new code, not by editing tested code.

**Trigger phrase:** "Adding a feature means writing a *new class*, not editing an old one. If you're adding another `if/else` to an existing method, you're violating OCP."

---

## Core Idea

You should be able to **extend** what a module does **without changing its source**. The goal: when requirements grow, you **add** new code rather than reopen stable, tested code — eliminating the risk of breaking what already works (regression).

The enablers are **abstraction + polymorphism**: program against an interface, and slot in new implementations.

---

## Real-life Analogy

Travel India → UK and your charger doesn't fit the socket. You don't **modify** the charger — you add a **travel adapter** that *extends* its usability. New capability, original untouched.

---

## Worked Example — Region-Based Tax in an Invoicing System

Tax differs by region (India GST 18%, US 8%, UK VAT 12%) and **new regions keep appearing.**

### ❌ Bad — violates OCP (every new region edits this method)
```java
class InvoiceProcessor {
    public double calculateTotal(String region, double amount) {
        if (region.equalsIgnoreCase("India"))   return amount + amount * 0.18;
        else if (region.equalsIgnoreCase("US")) return amount + amount * 0.08;
        else if (region.equalsIgnoreCase("UK")) return amount + amount * 0.12;
        else return amount;
    }
}
```
Adding Germany means **editing tested code** → risks breaking existing logic, hard to test/scale.

### ✅ Good — follows OCP (new region = new class)
```java
interface TaxCalculator {                       // the abstraction
    double calculateTax(double amount);
}

class IndiaTaxCalculator implements TaxCalculator {
    public double calculateTax(double amount) { return amount * 0.18; } // GST
}
class USTaxCalculator implements TaxCalculator {
    public double calculateTax(double amount) { return amount * 0.08; } // Sales Tax
}
class UKTaxCalculator implements TaxCalculator {
    public double calculateTax(double amount) { return amount * 0.12; } // VAT
}

class Invoice {                                  // depends on abstraction, injected in
    private double amount;
    private TaxCalculator taxCalculator;
    public Invoice(double amount, TaxCalculator taxCalculator) {
        this.amount = amount;
        this.taxCalculator = taxCalculator;
    }
    public double getTotalAmount() {
        return amount + taxCalculator.calculateTax(amount);
    }
}
```
**Adding Germany now touches nothing existing:**
```java
class GermanyTaxCalculator implements TaxCalculator {
    public double calculateTax(double amount) { return amount * 0.15; }
}
// new Invoice(amount, new GermanyTaxCalculator());  ← that's the whole change
```

**How it works:** interface = contract → concrete calculators encapsulate each region's rule → `Invoice` is decoupled via **Dependency Injection** (receives the calculator from outside) → `main` wires the right one in. (Notice this is literally the **Strategy pattern**, and it leans on **DIP** — OCP and DIP travel together.)

## When to Apply OCP

- A module is **expected to evolve** with shifting business/technical requirements.
- You need to **extend without touching tested code**.
- Building **frameworks, plugins, or extensible systems** (billing engines, tax calculators, UI widgets).
- Protecting **production-ready modules** from regression.
- A class is becoming a **God Class** (too many responsibilities/branches) → extract behaviors into extendable components.

> ⚠️ **Don't apply preemptively.** Adding abstraction with no real extension need just creates complexity. OCP is most effective **in response to observed patterns of change** — wait until you see the axis of variation.

---

## Common Misconceptions

- *"OCP means code is never changed again."* — No. It means avoid changing **core logic**; behavior is extended safely. Bug fixes still happen.
- *"Too many classes, it's overkill."* — Yes, more classes/interfaces, but the trade buys modularity, testability, maintainability for evolving systems.
- *"OCP makes code harder to read."* — In tiny/short-lived projects, maybe. In complex/changing systems, removing big `if/else` chains improves clarity.
- *"Apply it upfront, always."* — No, that risks needless abstraction; apply on emerging change patterns.
- *"Refactoring contradicts OCP."* — Opposite: refactoring is often the **step toward** OCP compliance.
- *"OCP means no retesting legacy."* — New extensions still need thorough testing for correctness + integration.

---

## Pitfalls / Things to Remember

- The tell-tale violation: **a growing `if/else`/`switch` over a type** inside one method → replace with polymorphism.
- OCP needs a stable **abstraction**; if the interface itself keeps changing, you don't have OCP.
- Balance with **YAGNI** — abstract for the variation you actually have, not every imaginable one.
