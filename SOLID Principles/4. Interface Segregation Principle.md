# Interface Segregation Principle (ISP) — the **I** in SOLID

> **One-liner:** **Don't force a class to depend on methods it doesn't use** — prefer many small, role-specific interfaces over one fat one.

**Trigger phrase:** "If a class implements a method with an empty body or 'not needed' comment, the interface is too fat. Split it by client role."

---

## Core Idea

A "fat" interface bundles unrelated capabilities, so any class implementing it is **forced to provide methods it has no use for** (usually as empty/dummy stubs). ISP says: **split interfaces by the needs of each client** so a class depends only on what it actually uses.

> ISP is SRP applied to **interfaces** — each interface should serve **one role/client**.

---

## Understanding — the Uber rider

Order an Uber as a **rider**: you care about *booking rides, tracking the driver, paying*. You do **not** care about *accepting rides, verifying licenses, or tracking earnings* — that's the **driver's** world. One giant interface exposing both roles would be confusing and wrong.

---

## Worked Example — Uber App Interfaces

### ❌ Bad — one fat interface (violates ISP)
```java
interface UberUser {
    void bookRide();
    void acceptRide();
    void trackEarnings();
    void ratePassenger();
    void rateDriver();
}

class Rider implements UberUser {
    public void bookRide()      { /* yes */ }
    public void acceptRide()    { /* not needed */ }   // forced stub
    public void trackEarnings() { /* not needed */ }   // forced stub
    public void ratePassenger() { /* not needed */ }   // forced stub
    public void rateDriver()    { /* yes */ }
}
```
`Rider` is dragged into implementing driver-only methods → messy, misleading, fragile.

### ✅ Good — segregated by role (follows ISP)
```java
interface RiderInterface {
    void bookRide();
    void rateDriver();
}
interface DriverInterface {
    void acceptRide();
    void trackEarnings();
    void ratePassenger();
}

class Rider implements RiderInterface {
    public void bookRide()   { /* yes */ }
    public void rateDriver() { /* yes */ }
}
class Driver implements DriverInterface {
    public void acceptRide()    { /* yes */ }
    public void trackEarnings() { /* yes */ }
    public void ratePassenger() { /* yes */ }
}
```
Each class implements **exactly what it needs — no more, no less.** A class can still implement **multiple** small interfaces if a role genuinely needs them (e.g. a future `DeliveryPartner`).

---

## Benefits

- **Cleaner codebase** — no classes bloated with irrelevant methods.
- **Flexibility** — change one role's interface without touching others.
- **Maintainability/testability** — small interfaces are easy to understand and mock.
- **Fewer bugs** — no accidental use/override of methods a class shouldn't have.
- **Scalability** — new roles (e.g. Uber Eats delivery partners) slot in via new interfaces.

---

## When to Apply ISP

- A class implements methods it **doesn't use** (empty/dummy bodies).
- An interface is **growing too big** and serves multiple unrelated client types.
- Adding a feature forces edits across **several unrelated classes**.
- Designing **APIs/plugins** where exposing only relevant methods improves usability.

---

## Pitfalls / Things to Remember

- **The smell is the empty stub.** `// not needed` in an override is the loudest ISP violation signal.
- **Don't over-segregate** into one-method interfaces everywhere — group by *cohesive role*, not by individual method.
- ISP is about the **client's** dependency surface; design interfaces from the **consumer's** point of view, not the implementer's.
- Memory hook: **Fat interfaces bad; slim, purpose-specific interfaces good.**
