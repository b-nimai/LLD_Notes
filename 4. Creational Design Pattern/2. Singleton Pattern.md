# Singleton Pattern

> **One-liner:** Ensure a class has **exactly one instance** and give the whole app a **single global access point** to it.

**Trigger phrase:** "I need *one shared object* for the whole app's lifetime (logger / config / DB connection pool / cache) → **private constructor + static instance + static `getInstance()`**. The catch is always **thread safety**."

> **Why it's Creational:** it controls *how the object is created* — `getInstance()` returns the **existing** instance instead of letting `new` make a fresh one each time.

---

## Core Idea

You want a class where creating multiple objects causes redundancy, wasted memory, or inconsistent behavior — **logging, configuration, DB connections, caches**. Singleton **restricts creation to one instance** and routes everyone through the same object.

### Real-life analogy
**Print spooler.** Many computers send jobs to one printer; if each talked to the printer directly, jobs would jumble or crash it. A single shared **spooler** queues every job through one centralized instance.

### The 3 mechanics
1. **Private constructor** — blocks `new` from outside.
2. **Static instance variable** — holds the one object.
3. **Public static `getInstance()`** — the global access point; always returns the same object.

---

## Eager vs Lazy

| | Eager Loading | Lazy Loading |
|---|---------------|--------------|
| **When created** | at class load, always | on first `getInstance()` call |
| **Analogy** | fire extinguisher (always there, just in case) | coffee machine (brews only when asked) |
| **Thread-safe?** | ✅ inherently | ❌ **not** by default |
| **Trade-off** | wastes memory if never used (bad for heavy objects) | saves memory; needs synchronization |

```java
// Eager — instance built when class loads
class EagerSingleton {
    private static final EagerSingleton instance = new EagerSingleton();
    private EagerSingleton() {}                 // private → no external `new`
    public static EagerSingleton getInstance() { return instance; }
}

// Lazy — built on first request (NOT thread-safe!)
class LazySingleton {
    private static LazySingleton instance;
    private LazySingleton() {}
    public static LazySingleton getInstance() {
        if (instance == null) instance = new LazySingleton();  // race window here
        return instance;
    }
}
```

## ⭐ Thread Safety — the heart of the interview

> **The bug:** two threads hit `getInstance()` first-time simultaneously, **both pass the `null` check**, both call `new` → **two instances**, Singleton broken. It's intermittent, severe, and costly (corrupts shared resources).

Four ways to fix it, cheapest-to-best:

### 1. Synchronized method
```java
public static synchronized Singleton getInstance() {
    if (instance == null) instance = new Singleton();
    return instance;
}
```
Only one thread in the method at a time. **Simple, but** every call pays the lock cost *forever*, even after creation → slow under high concurrency.

### 2. Double-Checked Locking (DCL)
```java
private static volatile Singleton instance;   // volatile is mandatory
public static Singleton getInstance() {
    if (instance == null) {                    // 1st check: no lock once created
        synchronized (Singleton.class) {
            if (instance == null)              // 2nd check: only one creates
                instance = new Singleton();
        }
    }
    return instance;
}
```
Lock **only on first creation**; afterwards lock-free. `volatile` ⇒ all threads read the *latest* instance from main memory (without it, a half-constructed object can leak). Needs Java 1.5+.

### 3. Bill Pugh (static inner holder) — **best practice**
```java
public class Singleton {
    private Singleton() {}
    private static class Holder { static final Singleton INSTANCE = new Singleton(); }
    public static Singleton getInstance() { return Holder.INSTANCE; }
}
```
`Holder` isn't loaded until `getInstance()` is first called → **lazy + thread-safe + no `synchronized`/`volatile`**, thanks to the JVM's class-loading guarantees. Cleanest of all.

### 4. Eager loading
Sidesteps threading entirely by building upfront — valid, but may waste memory.

> **Pick order:** **Bill Pugh** by default → **DCL** if you need explicit lazy control → **synchronized** only if simplicity beats performance → **eager** for cheap, always-used objects.

## Pros & Cons

**Pros:** guarantees one instance · global access to a shared resource/config · supports lazy loading · clean centralized control.

**Cons (interview gold):**
- **Hides global state** → hard to **unit test / mock** (state leaks between tests).
- **Tight coupling** — every consumer is bound to the concrete Singleton.
- **Violates SRP** — it manages *both* its own lifecycle *and* its real job.
- **Race conditions** if threading is handled carelessly.
- Confused with **Factory** when it takes construction parameters.

> **The modern alternative:** prefer **Dependency Injection** — let the container create one instance and inject it. You get "one instance" *without* the global state, so testing stays easy. (This is [[5. Dependency Inversion Principle]] in action.)

---

## Pitfalls / Things to Remember

- **Lazy ≠ thread-safe.** The plain `if (instance == null)` is the classic race — never ship it as-is in multi-threaded code.
- **DCL without `volatile` is broken.** Don't drop it — instruction reordering can expose a half-built object.
- **Don't reach for Singleton reflexively** — global state is a smell; if DI is available, use it.
- **Singleton vs static class:** Singleton is a real object (can implement interfaces, be passed around, lazy-loaded); a static class can't. Prefer Singleton when you need polymorphism.
