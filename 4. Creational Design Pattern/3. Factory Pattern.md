# Factory Pattern

> **One-liner:** Move object creation behind a **factory method** — the client asks the factory for an object *by input/condition* instead of calling `new` on a concrete class itself.

**Trigger phrase:** "I'm doing `new ConcreteClass()` inside business logic, picked by an `if/else` over a type string → a growing `switch` that breaks **OCP** every time I add a type. **Centralize creation in a factory.**"

> **Load-bearing idea:** the factory **hides the instantiation logic** from the client. The client codes to an **interface** and never learns which concrete class it got.

---

## Core Idea

The **Factory Pattern** provides an interface for **creating objects**, deciding the concrete type **at runtime** based on input. Reach for it when:
- the client must work with **multiple types** of an object,
- **which class to instantiate is decided at runtime**,
- creation is **complex or needs central control**.

### Real-life analogy
**Ordering pizza.** You say "I'd like a Margherita." You don't go into the kitchen and assemble it — the **kitchen (factory)** builds the specific pizza and hands it over. You (client) don't know or care about the ingredients/classes used.

### Structure (3 parts)
1. **Product** — interface/abstract class defining what the product does (`Shape`).
2. **Concrete Products** — implementations (`Circle`, `Square`).
3. **Factory** — a method returning the right concrete product from input.

```java
interface Shape { void draw(); }
class Circle implements Shape { public void draw(){ System.out.println("Circle"); } }
class Square implements Shape { public void draw(){ System.out.println("Square"); } }

class ShapeFactory {
    public Shape getShape(String type) {
        if (type.equalsIgnoreCase("CIRCLE")) return new Circle();
        if (type.equalsIgnoreCase("SQUARE")) return new Square();
        throw new IllegalArgumentException("Unknown shape: " + type);
    }
}
// client: Shape s = factory.getShape("CIRCLE"); s.draw();  // never sees `new Circle()`
```

## ⭐ Worked example — Logistics (the OCP payoff)

**❌ Without Factory** — creation tangled into business logic:
```java
class LogisticsService {
    public void send(String mode) {
        if (mode.equals("Air"))  { Logistics l = new Air();  l.send(); }
        else if (mode.equals("Road")) { Logistics l = new Road(); l.send(); }
    }
}
```
Problems: **tight coupling** to `Air`/`Road`, adding `Ship` **modifies** this method (**breaks OCP**), creation + business logic mixed, duplication, hard to test.

**✅ With Factory** — creation pulled out:
```java
class LogisticsFactory {
    public static Logistics getLogistics(String mode) {
        if (mode.equalsIgnoreCase("Air"))  return new Air();
        if (mode.equalsIgnoreCase("Road")) return new Road();
        throw new IllegalArgumentException("Unknown mode: " + mode);
    }
}
class LogisticsService {
    public void send(String mode) {
        Logistics l = LogisticsFactory.getLogistics(mode);  // service only does business logic
        l.send();
    }
}
```
Now adding **Ship** touches **only the factory**, never the service.

## Pros & Cons

**Pros:**
- **Loose coupling** — client codes to the interface, not concrete classes.
- **OCP / extensibility** — add a new type without touching client code (only the factory).
- **SRP** — creation centralized in one place; business logic stays clean.
- **Runtime flexibility** — "which object" deferred to input/config.
- **Reuse** — one place for instantiation logic, no duplication.

**Cons:**
- **Extra complexity / boilerplate** — more classes & an interface; overkill for tiny programs.

---

## Family / Related

- **Factory Method vs Abstract Factory:** *Factory Method* = one method picks **one product**; *Abstract Factory* = a factory that creates a **family of related products**. (Both GoF Creational.)
- The "good" refactor is really **OCP + DIP** applied to creation → see [[2. Open Closed Principle]] and [[5. Dependency Inversion Principle]].
- Sibling creational pattern: [[2. Singleton Pattern]] (often the factory *is* a singleton).

---

## Pitfalls / Things to Remember

- **The factory's own `if/else` still grows.** Factory doesn't *delete* the type switch — it **centralizes** it to one spot so the rest of the code stays closed. For full OCP, register types in a `Map<String, Supplier<Logistics>>`.
- **Don't over-apply.** Two object types you'll never extend? A plain `new` is fine — a factory is needless ceremony (the Cons are real).
- **Return the interface, not the concrete type** — leaking `Circle` instead of `Shape` re-couples the client.
- **Throw, don't return `null`,** on unknown input — a `null` here turns into a `NullPointerException` far from the cause.
