# Builder Pattern

> **One-liner:** Build a complex object **step by step** through a fluent API, then call `.build()` — instead of cramming every field into one giant constructor.

**Trigger phrase:** "My constructor has **many params, most of them optional** — callers pass `null, null, false` just to skip them, and I'm spawning **overloaded constructors** for every combo (telescoping). **Separate construction from the object: a Builder.**"

> **Load-bearing idea:** required fields go in the **Builder's constructor**; optional fields are set via chained `withX()` methods that **return `this`**; `build()` produces an **immutable** object via the target's **private constructor**.

---

## Core Idea

The **Builder Pattern** separates the *construction* of a complex object from its *representation*, so the **same step-by-step process** can produce different configurations. Reach for it when an object has **many fields, several optional**, and you want **readable, safe, immutable** creation.

### Real-life analogy
**Custom burger / pizza order.** You pick bun, patty, toppings, cheese, sides — one choice at a time. The kitchen assembles it from your selections. Different customers run the **same process** to get totally different meals.

### The problem it kills — Telescoping Constructor anti-pattern
```java
class BurgerMeal {
    BurgerMeal(String bun, String patty) { ... }
    BurgerMeal(String bun, String patty, boolean cheese) { ... }
    BurgerMeal(String bun, String patty, boolean cheese, String side) { ... }
    BurgerMeal(String bun, String patty, boolean cheese, String side, String drink) { ... }
}
// caller: new BurgerMeal("wheat","veg", null, null, false);  // what do these args even mean?
```
Hard to read, error-prone arg order, `null` clutter, **doesn't scale** as fields grow. Java has no default/named params, so devs pile on overloads — exactly what Builder fixes.

## ⭐ Canonical solution

```java
class BurgerMeal {
    // all final → immutable once built
    private final String bunType, patty;        // required
    private final boolean hasCheese;            // optional
    private final List<String> toppings;
    private final String side, drink;

    private BurgerMeal(BurgerBuilder b) {       // private → only Builder can construct
        this.bunType = b.bunType;  this.patty = b.patty;
        this.hasCheese = b.hasCheese;  this.toppings = b.toppings;
        this.side = b.side;  this.drink = b.drink;
    }

    static class BurgerBuilder {
        private final String bunType, patty;    // required → in builder's constructor
        private boolean hasCheese;              // optional → defaulted
        private List<String> toppings;
        private String side, drink;

        BurgerBuilder(String bunType, String patty) {   // force required fields up front
            this.bunType = bunType;  this.patty = patty;
        }
        BurgerBuilder withCheese(boolean c)        { this.hasCheese = c; return this; }   // return this
        BurgerBuilder withToppings(List<String> t) { this.toppings = t; return this; }     // → chaining
        BurgerBuilder withSide(String s)           { this.side = s; return this; }
        BurgerBuilder withDrink(String d)          { this.drink = d; return this; }
        BurgerMeal build()                         { return new BurgerMeal(this); }
    }
}

// usage — fluent, self-documenting, set only what you need:
BurgerMeal plain  = new BurgerMeal.BurgerBuilder("wheat","veg").build();
BurgerMeal loaded = new BurgerMeal.BurgerBuilder("multigrain","chicken")
                        .withCheese(true).withToppings(List.of("lettuce","onion"))
                        .withSide("fries").withDrink("coke").build();
```

**Four mechanics:** ① **private constructor** (only the builder builds) · ② **static nested Builder** mirrors the fields · ③ **fluent `withX()` returns `this`** → method chaining · ④ **`build()`** finalizes into an immutable object.

---

## Pros & Cons

**Pros:**
- **Kills telescoping** — no overloaded-constructor cascade.
- **Immutability** — all fields `final`, set once at `build()` → thread-safe.
- **Readable / expressive** — `withCheese(true)` beats a bare `true` in arg position 3.
- **Flexible** — configure only the fields you care about; great for config/domain objects.

**Cons:**
- **Boilerplate** — a whole second (builder) class to maintain.
- **Overkill for 1–2 fields** — a plain constructor is simpler; don't add ceremony.

- Sibling GoF creational pattern: [[3. Factory Pattern]] / [[5. Abstract Factory Pattern]] — Factory chooses **which** object; Builder controls **how** one complex object is assembled.
- **Builder vs Factory:** Factory returns a product in *one call* by type; Builder assembles a product *over many calls* by part.
- Immutability + final fields tie back to clean class design — see [[2. Singleton Pattern]] (also controls construction via a private constructor).
- Real-world: **Lombok `@Builder`** auto-generates this; `StringBuilder`, `Stream.Builder`, Amazon cart-item construction.

---

## Pitfalls / Things to Remember

- **Required vs optional split is the whole point** — required → builder constructor (can't forget them); optional → `withX()`. Putting required fields in setters lets callers build an invalid object.
- **Every `withX()` must `return this`** — forget it and the chain breaks (returns `void`).
- **Make the product immutable** — `final` fields + private constructor; otherwise you lose the thread-safety benefit and it's just setters in disguise.
- **Don't reach for it on small classes** — 1–2 fields = constructor; Builder is for *many, mostly optional* fields.
