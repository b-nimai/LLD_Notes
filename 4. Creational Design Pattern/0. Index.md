# Creational Design Patterns — Index

30-second whole-topic recall. Read the **Trigger** and try to recall the **30-sec revision** before opening the note.

**Creational patterns** = the **5 GoF patterns about *how objects are created*** (Singleton · Factory Method · Abstract Factory · Builder · Prototype) — abstract the instantiation so the system doesn't depend on it.

| # | Topic | Trigger phrase | 30-sec revision |
|---|-------|----------------|-----------------|
| 1 | [1. Introduction](1.%20Introduction.md) | "Pattern = a *recipe*, not the dish; 23 GoF patterns in 3 buckets — **Creational = make, Structural = compose, Behavioral = interact**" | **Design pattern** = time-tested, reusable **blueprint** (not copy-paste code) for a recurring design problem → don't reinvent the wheel + shared vocabulary. Analogy: cooking **recipe**. Formalized by **Gang of Four (1994)** → **23 patterns, 3 categories**: **Creational** (object *creation* — vending machine hides how the drink is made → Factory) · **Structural** (object *composition* — USB-C↔micro-USB **Adapter**) · **Behavioral** (object *interaction/responsibility* — restaurant waiter = **Mediator**). Creational(5): Singleton, Factory Method, Abstract Factory, Builder, Prototype. Pitfall: a pattern is a blueprint not code; don't force one where there's no recurring problem; patterns lean on SOLID. |
| 2 | [2. Singleton Pattern](2.%20Singleton%20Pattern.md) | "Need *one shared object* for the whole app (logger/config/DB) → private ctor + static instance + static `getInstance()`; catch = **thread safety**" | **Singleton** = exactly **one instance** + **global access point**. Creational because `getInstance()` returns the *existing* object, not a new one. Mechanics: **private constructor** (block `new`) + **static instance** + **static getInstance()**. Analogy: print **spooler**. **Eager** (built at class load, thread-safe, may waste memory) vs **Lazy** (built on first call, saves memory, **NOT thread-safe**). ⭐ Thread safety: two threads pass the `null` check → two instances. Fixes: **synchronized method** (simple, slow forever) → **Double-Checked Locking** + `volatile` (lock only on first create) → **Bill Pugh** static inner holder (lazy + thread-safe, no sync/volatile — **best**) → eager. Cons: hidden global state → hard to test/mock, tight coupling, **violates SRP**, race conditions. Modern alt: **Dependency Injection**. |
| 3 | [3. Factory Pattern](3.%20Factory%20Pattern.md) | "Doing `new ConcreteClass()` in business logic via an `if/else` over a type → growing switch breaks **OCP** → centralize creation in a factory" | **Factory** = interface for creating objects; concrete type chosen **at runtime** by input, instantiation **hidden** from client (codes to the interface). Analogy: order **pizza**, kitchen builds it. Structure: **Product** (interface, e.g. `Shape`/`Logistics`) + **Concrete Products** (`Circle`/`Square`, `Air`/`Road`) + **Factory** (`getShape`/`getLogistics` returns by input). Payoff: `LogisticsService` calling `new Air()`/`new Road()` in `send()` = tight coupling + adding `Ship` edits the method (breaks OCP); move creation into `LogisticsFactory` → service does only business logic, new mode = touch factory only. Pros: loose coupling, **OCP**, **SRP**, runtime flexibility, reuse. Cons: extra classes/boilerplate. Pitfall: factory's own `if/else` still grows (centralized, not gone → use a `Map<String,Supplier>` for true OCP); return the interface not concrete; throw on unknown, don't return null; don't over-apply. |
| 4 | [4. Builder Pattern](4.%20Builder%20Pattern.md) | "Constructor has **many params, mostly optional** — callers pass `null,null,false`; spawning overloaded constructors (telescoping) → build step-by-step via a **Builder**" | **Builder** = construct a complex object **step by step**, then `.build()`. Kills the **telescoping-constructor** anti-pattern (one overload per optional field). Analogy: **custom burger/pizza** order. 4 mechanics: ① **private constructor** (only builder builds) ② **static nested Builder** mirrors fields ③ **required → builder's constructor**, **optional → `withX()` that `return this`** (fluent chaining) ④ **`build()`** → **immutable** object (all fields `final`). `new BurgerMeal.BurgerBuilder("wheat","veg").withCheese(true).withSide("fries").build()`. Pros: no telescoping, immutability/thread-safe, readable, set only what you need. Cons: boilerplate (extra class), overkill for 1–2 fields. **vs Factory:** Factory picks *which* object; Builder controls *how* one complex object is assembled. Real-world: Lombok `@Builder`, `StringBuilder`. Pitfall: required-vs-optional split is the point; every `withX()` must `return this`; make product immutable. |
| 5 | [5. Abstract Factory Pattern](5.%20Abstract%20Factory%20Pattern.md) | "Need a **matching set** that must stay consistent (gateway **+** its invoice; button **+** checkbox), and which set depends on **context** (region/theme/OS) → **one factory per family**" | **Abstract Factory** = **factory of factories**; interface to create a **family of related products** that work together, without naming concretes. Analogy: **regional checkout kit** — IndiaFactory → Razorpay/PayU **+** GSTInvoice; USFactory → PayPal/Stripe **+** USInvoice. Structure: `RegionFactory` interface with `createPaymentGateway()` + `createInvoice()`; concrete factories build **one consistent family**; `CheckoutService` depends only on the **abstractions** (OCP+DIP). New region = add a factory, service unchanged. Pros: **family consistency**, encapsulated creation, scalable, testable. Cons: **adding a new *product* to the family is hard** (touch every factory), heavy boilerplate, variant fixed at wire-time. **vs Factory:** Factory = one product; Abstract Factory = a whole family (often built from several factory methods). Pitfall: it's about *families* not one object; easy to add families, hard to add members; return interfaces, throw on unknown. |
| 6 | [6. Prototype Pattern](6.%20Prototype%20Pattern.md) | "Setup is **expensive/complex** and I need **many near-identical objects** with small tweaks → `new X()` + pile of setters each time → **clone a prototype**" | **Prototype** = create objects by **cloning a pre-configured one** (`clone()`), copy + tweak a few fields. Object copies **itself** → client decoupled from concrete class + skips costly init. Analogy: **photocopy** machine (write once, copy, change the name). 3 mechanics: ① prototype interface with `clone()` ② concretes implement self-copy ③ **registry** stores prototypes, hands out **clones** (never the original). ⭐ **Deep vs Shallow:** `Object.clone()` is **shallow** (shares references → mutating nested state corrupts the original); Prototype usually wants a **deep** copy (fully independent). Pros: faster creation, less subclassing, runtime tweak, decoupled. Cons: deep clone is hard, circular refs tricky, bug-prone shallow copies. **vs Factory:** Factory constructs new; Prototype copies an existing configured one. Pitfall: registry must return a *clone*; implement `Cloneable` (else `CloneNotSupportedException`); watch circular refs. |
