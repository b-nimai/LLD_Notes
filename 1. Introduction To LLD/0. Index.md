| 1 | [1. Introduction](1.%20Introduction.md) | "HLD = architect's blueprint (where rooms go); LLD = wiring/plumbing (exact switches, materials) — detailed planning *before* writing code" | **LLD** = the **code-level blueprint** that turns architecture into implementable detail (classes, methods, fields, algorithms, data structures); the **bridge between HLD and actual code**. House analogy: **HLD = what pieces exist + how they connect** (abstract: modules, interfaces, system diagrams); **LLD = how each piece is built** (detailed: class/method diagrams). Example: login box at HLD → `login()/signUp()/forgotPassword()` with behavior at LLD. **Characteristics:** granular/code-level, implementation-focused (pseudocode + flow/sequence diagrams), heavy **OOP** (e.g. `Notification`→`Email/SMSNotification`). Stakeholders = senior devs/tech leads/managers. **Why:** avoids rework, improves collaboration, scalability, best practices — and it's the core of **senior interviews**. Pitfall: don't confuse altitudes (HLD=what/where, LLD=how); LLD is the *plan for* code, not the coding itself; needs strong OOP. |
| 2 | [2. Software Design Principles](2.%20Software%20Design%20Principles.md) | "DRY = one source of truth (no copy-paste logic) · KISS = simplest thing that works (no clever code) · YAGNI = build only when actually needed (no speculative features)" | Three universal *mindset* guidelines (apply at HLD **and** LLD; broader than SOLID), each with a "when NOT to." **DRY — Don't Repeat Yourself:** every piece of *knowledge* has one authoritative representation → extract repeated logic into a method/utility (`AreaCalculator.calculateArea`); single point of change. Don't over-apply: **premature abstraction** (couples unrelated code — *rule of three*), performance-critical hot paths (indirection blocks inlining), readability cost, untested legacy. **KISS — Keep It Simple, Stupid:** simplest solution that *works*; no clever convoluted code (`return n % 2 == 0`, not an if/else with a temp var). ≠ simplistic — must still fully solve the problem. **YAGNI — You Aren't Gonna Need It:** implement only when *actually* needed, never on speculation (build create+view notes, not categories/tags/sync nobody asked for). Skip YAGNI when a requirement is **committed & imminent** (image support in 2 sprints) or perf-critical. Memory hook: DRY fights **duplication**, KISS fights **complexity**, YAGNI fights **speculation**; tie-breaker = **clarity wins**, all are guidelines not laws. |
