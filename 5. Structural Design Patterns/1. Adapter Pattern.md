# Adapter Pattern

> **One-liner:** A **wrapper** that converts one class's interface into the interface the client expects ‚Äî so two **incompatible interfaces** work together **without touching either class**.

**Trigger phrase:** "I have an existing/third-party class that does the right job but the **wrong method signature** (`makePayment` vs `pay`), and I can't/won't edit it ‚Üí **wrap it in an adapter** that implements my expected interface and delegates."

> **Load-bearing idea:** the adapter **implements the Target interface** and **holds the Adaptee**; its methods just **translate** the client's call into the adaptee's call. Client codes to the Target and never learns the adaptee exists.

---

## Core Idea

The **Adapter Pattern** lets classes with **incompatible interfaces** collaborate by inserting a **translator** in between. It bridges:
- **Target** ‚Äî the interface the **client expects** (`PaymentGateway.pay`).
- **Adaptee** ‚Äî the existing class with a **different** interface (`RazorpayAPI.makePayment`).
- **Adapter** ‚Äî implements Target, wraps an Adaptee, maps one call onto the other.

Reach for it when:
- you must use an existing class but its **interface doesn't match** what your system expects,
- you want to **reuse legacy code** without modifying it,
- you're **integrating a third-party API / external SDK**.

### Real-life analogy
**Travel plug adapter.** Your Indian charger won't fit a European socket. Instead of rewiring the charger *or* the socket, a **plug adapter** sits between them and makes them fit. Neither device changes.

### Structure (3 parts)
1. **Target interface** ‚Äî what the client calls (`PaymentGateway { pay(orderId, amount) }`).
2. **Adaptee** ‚Äî the incompatible existing class (`RazorpayAPI { makePayment(invoiceId, amountInRupees) }`).
3. **Adapter** ‚Äî `implements Target`, **has-a** Adaptee, translates the call.

```java
interface PaymentGateway { void pay(String orderId, double amount); }        // Target

class RazorpayAPI {                                                          // Adaptee (untouchable)
    public void makePayment(String invoiceId, double amountInRupees) { ... }
}

class RazorpayAdapter implements PaymentGateway {                            // Adapter
    private RazorpayAPI razorpayAPI = new RazorpayAPI();
    public void pay(String orderId, double amount) {
        razorpayAPI.makePayment(orderId, amount);   // translate pay() -> makePayment()
    }
}
```

## ‚≠ê Worked example ‚Äî Payment gateways (the integration payoff)

`CheckoutService` is written against one interface: `PaymentGateway.pay(orderId, amount)`.
`PayUGateway` already implements it ‚Üí drops right in. `RazorpayAPI` does the same job but exposes `makePayment(invoiceId, amountInRupees)` ‚Üí **can't be passed to `CheckoutService`**. This is **interface incompatibility**.

**‚ùå Without Adapter** ‚Äî Razorpay simply doesn't fit:
```java
new CheckoutService(new RazorpayAPI()); // ‚úó RazorpayAPI is not a PaymentGateway
```

**‚úÖ With Adapter** ‚Äî wrap Razorpay so it *looks like* a `PaymentGateway`:
```java
class RazorpayAdapter implements PaymentGateway {
    private RazorpayAPI razorpayAPI = new RazorpayAPI();
    @Override public void pay(String orderId, double amount) {
        razorpayAPI.makePayment(orderId, amount);   // adapt: pay -> makePayment
    }
}

// client is unchanged ‚Äî it still only knows PaymentGateway
CheckoutService service = new CheckoutService(new RazorpayAdapter());
service.checkout("12", 1780);
```
Neither `CheckoutService` nor `RazorpayAPI` was modified. Adding **Stripe/PayPal** = write one more adapter, client untouched ‚Üí **OCP**.

## Pros & Cons

**Pros:**
- **Reuse existing/legacy code** ‚Äî no changes to the adaptee.
- **OCP / extensibility** ‚Äî support a new provider by adding an adapter, not editing clients.
- **Minimal client changes** ‚Äî client keeps talking to the Target interface.
- **Simplifies third-party integration** ‚Äî isolates external APIs behind your interface.

**Cons:**
- **Extra layer of abstraction** ‚Äî one more class/indirection.
- **Overuse obscures design** ‚Äî a pile of adapters makes the architecture harder to follow.

---

## Real product use cases
- **Payment gateways** ‚Äî common `PaymentGateway`, one adapter per provider (Razorpay/PayU/Stripe/PayPal).
- **Logging** ‚Äî unify `log.debug(...)` over Log4j / SLF4J / `java.util.logging`.
- **Cloud SDKs** ‚Äî one storage interface adapting AWS S3 / GCS / Azure Blob for multi-cloud.

## Family / Related

- **Structural sibling ‚Äî [[2. UML Class Diagram]]** for the realize/dependency notations used above.
- **Adapter vs Facade:** Adapter **changes an interface** to match what the client expects (1 class); Facade **simplifies** a subsystem behind a new, easier interface (many classes) ‚Äî Facade isn't about compatibility.
- **Adapter vs Decorator:** both wrap, but Adapter **changes the interface**; Decorator **keeps the same interface** and **adds behavior**.
- **Adapter vs Proxy:** Proxy keeps the same interface and controls **access**; Adapter's whole point is a **different** interface.
- The refactor is **OCP + DIP** applied to integration ‚Üí [[2. Open Closed Principle]], [[5. Dependency Inversion Principle]].

---

## Pitfalls / Things to Remember

- **Adapter *implements the Target*, *composes the Adaptee*** ‚Äî get this backwards (extending the adaptee, implementing nothing) and the client still can't use it.
- **Adapter only translates** ‚Äî no business logic. If you're adding behavior, that's a **Decorator**, not an Adapter.
- **Don't "fix" it by editing the adaptee** ‚Äî you often *can't* (third-party jar), and even when you can it breaks other callers. The adapter exists precisely to avoid that.
- **Watch the mapping** ‚Äî signatures differ for a reason (`invoiceId` vs `orderId`, units/currency, response shapes). The adapter is where those mismatches get reconciled; a sloppy map is a silent bug.
- **Two flavors:** **object adapter** (composition ‚Äî *has-a* adaptee; preferred, shown here) vs **class adapter** (multiple inheritance ‚Äî *is-a* adaptee; impossible with single-inheritance interfaces in Java).
