# Facade Pattern

> **One-liner:** Put a **single simplified entry point** in front of a **complex subsystem** ã the client calls one high-level method, and the facade orchestrates all the messy internal service calls behind it.

**Trigger phrase:** "The client has to know **too much** ã it manually news up 5 services and calls them in the right order with the right params to do *one* logical operation. Wrap that whole workflow in **one facade method** (`bookTicket()`)."

> **Load-bearing idea:** the facade **doesn't change or add** anything to the subsystem ã it just **coordinates** it. The subsystem stays fully usable directly; the facade is a *convenience layer*, not a wall.

---

## Core Idea

The **Facade Pattern** provides a **unified, high-level interface** to a set of interfaces in a subsystem, making it easier to use. It's a **single door** into many rooms.

Reach for it when:
- a **subsystem is complex** ã many classes, many dependencies, a fixed call order,
- you want a **simpler API for the outside world**,
- you want to **reduce coupling** between client code and the subsystem's internals,
- you're **layering the architecture** (client ã facade ã subsystem).

### Real-life analogy
**Automatic vs manual car.** A manual car exposes the complexity ã clutch, gears, accelerator, all coordinated by you. An **automatic car is a facade**: you just pick "Drive / Reverse / Park" and the transmission handles the gear mechanics internally. Same engine underneath, simpler interface on top.

### Structure (3 parts)
1. **Subsystem classes** ã the many workers (`PaymentService`, `SeatReservationService`, `NotificationService`, `LoyaltyPointsService`, `TicketService`). They don't know the facade exists.
2. **Facade** ã holds references to the subsystem and exposes **one coarse-grained method** (`MovieBookingFacade.bookMovieTicket(...)`).
3. **Client** ã talks **only** to the facade.

## ãÙ Worked example ã BookMyShow ticket booking (the decoupling payoff)

**ã Without Facade** ã the client orchestrates everything itself:
```java
PaymentService paymentService = new PaymentService();
paymentService.makePayment("user123", 500);

SeatReservationService seatService = new SeatReservationService();
seatService.reserveSeat("movie456", "A10");

NotificationService notif = new NotificationService();
notif.sendBookingConfirmation("user@example.com");

LoyaltyPointsService loyalty = new LoyaltyPointsService();
loyalty.addPoints("user123", 50);

TicketService ticket = new TicketService();
ticket.generateTicket("movie456", "A10");
```
Problems: **tight coupling** (client knows all 5 classes), **duplicated** everywhere booking happens, the client **knows the correct order** (payment before ticketãÎ), and `Main` **violates SRP** ã it knows far too much.

**ã With Facade** ã one method hides the whole workflow:
```java
class MovieBookingFacade {
    private PaymentService paymentService = new PaymentService();
    private SeatReservationService seatReservationService = new SeatReservationService();
    private NotificationService notificationService = new NotificationService();
    private LoyaltyPointsService loyaltyPointsService = new LoyaltyPointsService();
    private TicketService ticketService = new TicketService();

    public void bookMovieTicket(String accountId, String movieId, String seatNumber,
                                String userEmail, double amount) {
        paymentService.makePayment(accountId, amount);          // orchestration +
        seatReservationService.reserveSeat(movieId, seatNumber); // correct order
        ticketService.generateTicket(movieId, seatNumber);
        loyaltyPointsService.addPoints(accountId, 50);
        notificationService.sendBookingConfirmation(userEmail);
        System.out.println("Movie ticket booking completed successfully!");
    }
}

// client ã knows one class, one method
new MovieBookingFacade().bookMovieTicket("user123", "movie456", "A10", "user@example.com", 500);
```
Client depends on **one** class. Change a service's internals or reorder steps ã edit the facade, **client untouched**.

## Pros & Cons

**Pros:**
- **Loose coupling** ã client depends on one facade, not N subsystem classes.
- **Simpler client** ã one call replaces a multi-step, ordered dance.
- **Flexibility** ã subsystem can evolve/reorganize without touching clients.
- **Layered architecture** ã clean client ã facade ã subsystem boundary.
- **Reuse / no duplication** ã the workflow lives in one place.

**Cons:**
- **Can become a god object** ã if it orchestrates too much, the facade itself **violates SRP**.
- **Hidden complexity** ã the subsystem's complexity still exists, just concealed; harder to see the full flow.
- **Extra indirection** ã one more layer to trace when debugging; subsystem errors are harder to diagnose through it.
- **Fragile if it changes often** ã a frequently-changing facade still ripples to clients.

## Real product use cases
- **BookMyShow / e-commerce checkout** ã one `placeOrder()` orchestrating payment, inventory, shipping, notification.
- **Spring `JdbcTemplate`** ã hides connection/statement/resultset/close boilerplate behind `query(...)`.
- **SLF4J / `javax` APIs, SDK client libraries** ã a friendly wrapper over a sprawling low-level API.

## Family / Related

- **Structural sibling ã [[1. Adapter Pattern]]:** Adapter **changes one class's interface** to match what a client expects (compatibility); Facade **wraps many classes** in a *new, simpler* interface (convenience). Adapter = make it *fit*; Facade = make it *simple*.
- **vs [[2. Decorator Pattern]]:** Decorator keeps the same interface and **adds behavior** to one object; Facade defines a **new** interface over **many** objects and adds no behavior.
- **vs Mediator (Behavioral):** both centralize interactions, but Facade is a **one-way simplifier** (clientãsubsystem, subsystem unaware); Mediator manages **two-way** peer communication.
- Enforces [[1. Single Responsibility Principle]] for the client and supports layered [[5. Dependency Inversion Principle]] designs.

---

## Pitfalls / Things to Remember

- **Facade adds no new behavior** ã if you're adding logic beyond orchestration, you're drifting toward a service/god object. Keep it a thin coordinator.
- **Don't seal off the subsystem** ã the facade is optional convenience; power users should still be able to call subsystem classes directly. It's a door, not a wall.
- **Watch the god-object smell** ã one facade doing payment + booking + refunds + reports = SRP violation. Split into multiple focused facades.
- **Facade ã  Adapter** ã the interview trap. Adapter is about **compatibility** (wrong signature ã right signature); Facade is about **simplicity** (many calls ã one call).
- **Errors get harder to trace** ã subsystem exceptions surface through the facade; make sure it doesn't swallow or obscure them.
