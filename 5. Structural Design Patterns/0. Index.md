# Structural Design Patterns — Index

30-second whole-topic recall. Read the **Trigger** and try to recall the **30-sec revision** before opening the note.

**Structural patterns** = the **7 GoF patterns about *how classes & objects are composed*** into larger structures (Adapter · Bridge · Composite · Decorator · Facade · Flyweight · Proxy) — assemble things flexibly without rigid coupling.

| # | Topic | Trigger phrase | 30-sec revision |
|---|-------|----------------|-----------------|
| 1 | [1. Adapter Pattern](1.%20Adapter%20Pattern.md) | "Existing/third-party class does the right job but has the **wrong method signature** (`makePayment` vs `pay`) and I can't edit it → **wrap it in an adapter** that implements my interface and delegates" | **Adapter** = a **wrapper** that converts one interface into the one the client expects, so **incompatible interfaces** work together **without modifying either class**. Analogy: **travel plug adapter** (charger and socket both unchanged). 3 parts: **Target** (interface client calls, `PaymentGateway.pay`) + **Adaptee** (existing class, wrong signature, `RazorpayAPI.makePayment`) + **Adapter** (`implements Target`, **has-a** Adaptee, **translates** the call). Payoff: `CheckoutService` codes to `PaymentGateway`; `PayUGateway` fits directly, `RazorpayAPI` doesn't → `RazorpayAdapter` wraps it and maps `pay()`→`makePayment()`; client & adaptee both untouched, new provider = new adapter (**OCP+DIP**). Real: payment gateways, logging (Log4j/SLF4J), cloud SDKs (S3/GCS). **vs Facade** (simplifies a subsystem, not compatibility) / **vs Decorator** (same interface, *adds* behavior) / **vs Proxy** (same interface, controls access). Pitfall: adapter *implements Target + composes Adaptee* (not the reverse); it only **translates**, no business logic; reconcile the signature/unit/currency mismatch carefully; prefer **object adapter** (composition) over class adapter. |
| 2 | [2. Decorator Pattern](2.%20Decorator%20Pattern.md) | "Many **combinations** of optional add-ons (toppings, text styles, I/O filters) and inheritance forces a class per combo (`CheeseOliveStuffedPizza`) → **class explosion** → compose with **wrappers**, not subclasses" | **Decorator** = **wrap** an object in another sharing its interface that **adds behavior**; stack wrappers to compose features **at runtime** instead of subclassing every combination. Analogy: **coffee** + milk + sugar + cream (each add-on wraps the drink). 4 parts: **Component** (interface `Pizza{getDescription,getCost}`) + **Concrete Component** (`MargheritaPizza`) + **Abstract Decorator** (`implements Pizza` **AND holds a `Pizza`** — the dual relationship, IS-A + HAS-A) + **Concrete Decorators** (`ExtraCheese`/`Olives`/`StuffedCrust`, each **delegates then adds**). ⭐ crux: decorator both *is* the interface (so it can be wrapped again → layers stack like a call stack) and *holds* one (to delegate to). Payoff: `n` toppings = `2^n` subclasses without it; with it = `n` decorators, `new StuffedCrust(new Olives(new ExtraCheese(new MargheritaPizza())))`, new topping = 1 class, base untouched (**OCP+SRP**). Real: Swiggy/Zomato add-ons, Docs bold/italic/underline, **Java I/O streams** (`BufferedReader(InputStreamReader(...))`). **vs Adapter** (changes interface, doesn't add behavior) / **vs Proxy** (controls access) / **vs Strategy** (swaps one algorithm vs stacks many layers). Pitfall: must implement interface AND hold ref; always **delegate then add** (else inner layers vanish); **order can matter** (tax-then-discount ≠ reverse); don't over-layer (ugly stack traces). |
| 3 | [3. Facade Pattern](3.%20Facade%20Pattern.md) | "Client news up 5 services and calls them in the **right order** with the right params to do **one** logical op → it knows too much (tight coupling, SRP violation) → hide the whole workflow behind **one facade method** (`bookTicket()`)" | **Facade** = a **single simplified entry point** in front of a **complex subsystem**; client calls one high-level method and the facade **orchestrates** the messy internal calls. Analogy: **automatic car** ("Drive/Reverse/Park" hides clutch+gears). 3 parts: **Subsystem classes** (many workers `PaymentService`/`SeatReservationService`/`Notification`/`Loyalty`/`TicketService`, unaware of facade) + **Facade** (`MovieBookingFacade`, has-a each service, exposes coarse `bookMovieTicket(...)`) + **Client** (talks only to facade). ⭐ crux: facade **adds no behavior**, only **coordinates** — subsystem still usable directly (it's a door, not a wall). Payoff: without it the client knows all 5 classes + correct order + duplicates it everywhere (breaks SRP); with it client depends on 1 class, internals change → edit facade only. Pure **composition + delegation, no interface/inheritance**. Real: BookMyShow/checkout `placeOrder()`, Spring `JdbcTemplate`, SDK client libs. **vs Adapter** (changes *one* interface for compatibility) / **vs Decorator** (same interface, adds behavior) / **vs Mediator** (two-way peer comms vs one-way simplifier). Pitfall: don't add logic (→ **god object**, SRP violation); don't seal off subsystem; Facade=simplicity, Adapter=compatibility (interview trap); errors harder to trace through the layer. |
