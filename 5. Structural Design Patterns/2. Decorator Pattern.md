# Decorator Pattern

> **One-liner:** **Wrap** an object in another object that shares its interface and **adds behavior** — stack wrappers to compose features at **runtime**, instead of building a subclass for every combination.

**Trigger phrase:** "I need many **combinations** of optional add-ons (toppings, text styles, I/O filters), and inheritance is forcing me to make a class per combination (`CheeseOliveStuffedPizza`) → **class explosion**. Compose with **wrappers**, not subclasses."

> **Load-bearing idea:** a decorator **implements the same interface** as the thing it wraps **and holds a reference to it**. Its method does `wrapped.method()` **plus its own bit**. Because it *is* the interface, another decorator can wrap it → **layers stack** like a call stack.

---

## Core Idea

The **Decorator Pattern** attaches new responsibilities to an **individual object, at runtime**, without changing its class or affecting other instances. Same interface in, richer behavior out.

Reach for it when:
- you need to **add responsibilities dynamically** (per-object, not per-class),
- you want to **avoid a subclass explosion** from feature combinations,
- you want each add-on to be a small, **reusable, composable** unit (**SRP + OCP**).

### Real-life analogy
**Coffee shop.** Order a base coffee, then *add milk, add sugar, add whipped cream*. Each add-on **wraps** the drink and adds cost/description — you never define a `MilkSugarWhippedCoffee` class.

### Structure (4 parts)
1. **Component** — interface everyone shares (`Pizza { getDescription(); getCost(); }`).
2. **Concrete Component** — the base object (`PlainPizza`, `MargheritaPizza`).
3. **Abstract Decorator** — `implements Component` **and holds a `Component`** (the `protected Pizza pizza`).
4. **Concrete Decorators** — extend the abstract decorator; each **delegates + adds** (`ExtraCheese`, `Olives`, `StuffedCrust`).

## ⭐ Worked example — Pizza toppings (the class-explosion payoff)

**❌ Without Decorator** — one subclass per combination, grows exponentially:
```java
class PlainPizza {}
class CheesePizza extends PlainPizza {}
class CheeseOlivePizza extends CheesePizza {}
class CheeseOliveStuffedPizza extends CheeseOlivePizza {}  // ...and on, and on
```
`n` toppings → up to `2^n` classes. Unmanageable, and reordering/removing a topping means yet another class.

**✅ With Decorator** — one class per topping, stack them freely:
```java
interface Pizza { String getDescription(); double getCost(); }        // Component

class MargheritaPizza implements Pizza {                              // Concrete Component
    public String getDescription() { return "Margherita Pizza"; }
    public double getCost() { return 200.0; }
}

abstract class PizzaDecorator implements Pizza {                      // Abstract Decorator
    protected Pizza pizza;                                            // ← wraps a Pizza
    public PizzaDecorator(Pizza pizza) { this.pizza = pizza; }
}

class ExtraCheese extends PizzaDecorator {                           // Concrete Decorator
    public ExtraCheese(Pizza pizza) { super(pizza); }
    public String getDescription() { return pizza.getDescription() + ", Extra Cheese"; }
    public double getCost() { return pizza.getCost() + 40.0; }       // delegate + add
}
// Olives (+30), StuffedCrust (+50) look identical in shape

// client — stack decorators at runtime, in any order
Pizza p = new MargheritaPizza();
p = new ExtraCheese(p);
p = new Olives(p);
p = new StuffedCrust(p);
p.getDescription();  // "Margherita Pizza, Extra Cheese, Olives, Stuffed Crust"
p.getCost();         // 200 + 40 + 30 + 50 = 320
```
New topping = **one new decorator**, zero edits to `Pizza`/`MargheritaPizza` → **OCP**. Adding `Ship`-style features never touches existing code.

### Class diagram (from the use-case)
```
                       «interface» Pizza
                       + getDescription(): String
                       + getCost(): double
             △  implements   △              ◆ composition (has-a)
    ┌────────┘               └────────┐          │
MargheritaPizza            PlainPizza  │   «abstract» PizzaDecorator
+getDescription()          +getDescription()  - pizza: Pizza
+getCost()                 +getCost()          + PizzaDecorator(pizza: Pizza)
                                                     △ inheritance
                                    ┌────────────────┼────────────────┐
                               ExtraCheese         Olives         StuffedCrust
                               +getDescription()   +getDescription()  +getDescription()
                               +getCost()          +getCost()         +getCost()
```
- **Two relationships at once — the crux:** `PizzaDecorator` **IS-A** `Pizza` (implements, so it fits anywhere a `Pizza` does) **and HAS-A** `Pizza` (the `pizza` field it wraps). Concrete decorators **inherit** from `PizzaDecorator`. (Notations: [[2. UML Class Diagram]].)

## Pros & Cons

**Pros:**
- **No subclass explosion** — one decorator per feature, not per combination.
- **OCP** — add behavior via new decorators without touching base classes.
- **Runtime flexibility** — add / remove / reorder features on a live object.
- **SRP** — each decorator owns exactly one add-on; composable & reusable.

**Cons:**
- **Many small classes** — one per feature can clutter the codebase.
- **Hard to debug** — deep wrapping means long, confusing stack traces.
- **Order can matter** — layering isn't always commutative; wrong order = wrong result.
- **Learning curve** — readers must follow the chaining to reason about behavior.

## Real product use cases
- **Food delivery (Swiggy/Zomato)** — base item + `CheeseDecorator`, `SauceDecorator` add-ons stack over it.
- **Text formatting (Docs/Word)** — `Underline(Bold(Italic(Text)))` layers styles without `BoldItalicUnderlineText` classes.
- **Java I/O streams** — the canonical example: `new BufferedReader(new InputStreamReader(inputStream))` — each wrapper adds buffering/decoding.

## Family / Related

- **Structural sibling — [[1. Adapter Pattern]]:** both **wrap** an object, but Adapter **changes the interface** (make incompatible fit); Decorator **keeps the same interface** and **adds behavior**. This is the #1 interview distinction.
- **vs Proxy:** Proxy also keeps the interface but controls **access** (lazy/cache/auth), not enrichment.
- **vs Inheritance:** Decorator is *composition at runtime*; inheritance is *fixed at compile time*. It's the concrete answer to "**favor composition over inheritance**."
- **vs Strategy:** Strategy swaps *one* pluggable algorithm; Decorator *stacks many* layers around an object.
- Enforces [[2. Open Closed Principle]] (add features, don't modify) and [[3. UML Class Diagram]] notations.

---

## Pitfalls / Things to Remember

- **The dual relationship is the whole trick:** the decorator must **implement the component interface AND hold a component reference**. Miss the "implements" and it can't be wrapped further; miss the "holds" and it has nothing to delegate to.
- **Always delegate, then add** — `return wrapped.getCost() + 40;`. Forgetting to call the wrapped object drops all inner layers silently.
- **Order can change the result** — tax-then-discount ≠ discount-then-tax. Decorators aren't guaranteed commutative; think about layering order.
- **Abstract decorators can have constructors** — they run on subclass instantiation to store the wrapped reference; that's expected, not a smell.
- **Don't over-layer** — deep chains are hard to debug (messy stack traces) and add call overhead; use it where combinations genuinely explode, not for one or two fixed features.
