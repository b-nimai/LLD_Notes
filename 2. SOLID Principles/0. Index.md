# SOLID Principles — Index

30-second whole-topic recall. Read the **Trigger** and try to recall the **30-sec revision** before opening the note.

**SOLID** = five OOP design principles (Robert C. Martin) for clean, scalable, maintainable code. **S**RP · **O**CP · **L**SP · **I**SP · **D**IP. O/L/I/D all lean on **abstraction + polymorphism**.

| # | Topic | Trigger phrase | 30-sec revision |
|---|-------|----------------|-----------------|
| 1 | [1. Single Responsibility (SRP)](1.%20Single%20Responsibility%20Principle.md) | "One class, one job — if you describe a class with 'AND', split it; a class should have one *reason to change*" | A class has **one responsibility = one reason to change**; bundling many **couples** them (change one → ripple). Analogy: a chef who also cleans/serves/orders does all poorly → split roles. Example: a god `TUFplusCompiler` → `DriverCodeGenerator`/`SyntaxChecker`/`TestRunner`/`DatabaseManager`/`UserOutputHandler` + a `Coordinator`. Wins: maintainability, readability, reuse, testability, low change-risk. Smells: DB logic mixed with business logic, UI coupled to logic. Pitfall: **not** "one method per class" (it's one reason to change); don't over-split; applies to **methods/modules/microservices/systems**, not just classes. |
| 2 | [2. Open/Closed (OCP)](2.%20Open%20Closed%20Principle.md) | "Open for extension, closed for modification — add a *new class*, don't edit tested code; a growing `if/else` over a type = violation" | Extend behavior **without changing source** → no regression on stable code; enabled by **abstraction + polymorphism**. Analogy: travel **adapter** extends a charger without modifying it. Example: region tax via `if/else` chain (bad) → `TaxCalculator` interface + `India/US/UK...TaxCalculator`, injected into `Invoice`; adding Germany = new class, edit nothing (this is **Strategy** + leans on DIP). Apply when a module is expected to evolve / building plugins / a God Class is forming — but **not preemptively** (needless abstraction). Misconceptions: ≠ "never change code", more classes is an OK trade, refactoring *helps* OCP. Pitfall: tell-tale = growing `switch`/`if` over a type → use polymorphism. |
| 3 | [3. Liskov Substitution (LSP)](3.%20Liskov%20Substitution%20Principle.md) | "A subclass must be a drop-in for its parent — swapping the child in must not surprise the caller (wrong result / exception / do-nothing)" | If `S` subtype of `T`, replace `T` with `S` **anywhere** without breaking correctness; subclass must **honor the parent's contract**. Analogy: pet hotel assumes pets can be fed/walked/groomed → a **snake** breaks it (violation), a **hamster** fits (valid). Classic violation: `Square extends Rectangle` overrides `setWidth/setHeight` to keep sides equal → `printArea` expects 50, gets 100 (broke the independent-sides invariant). Good: `Email/TextNotification extends Notification` — caller unchanged. Spot it: override changes meaning, throws unexpected, returns null/sentinel, **weakens precondition / strengthens postcondition**. Pitfall: English "is-a" ≠ valid subtype (model by **contract**); fails at **runtime** not compile; fix = restructure hierarchy; prefer composition. |
| 4 | [4. Interface Segregation (ISP)](4.%20Interface%20Segregation%20Principle.md) | "Don't force a class to implement methods it doesn't use — split fat interfaces by client role; an empty-body override = the smell" | Many small **role-specific** interfaces > one fat one; a fat interface forces classes into **dummy stubs** for methods they don't need (ISP = SRP for interfaces). Example: fat `UberUser` (book/accept/earnings/rate...) forces `Rider` to stub driver methods → split into `RiderInterface` + `DriverInterface`; each class implements only what it needs (and can implement **multiple** small ones). Wins: cleaner code, flexibility, testable/mockable, fewer bugs, scalable (Uber Eats delivery role slots in). Apply when a class stubs unused methods / interface grows / serves many client types. Pitfall: the **empty `// not needed` stub** is the loudest signal; don't over-segregate to one-method interfaces — group by **cohesive role**; design from the **client's** view. |
| 5 | [5. Dependency Inversion (DIP)](5.%20Dependency%20Inversion%20Principle.md) | "Depend on abstractions, not concretions — don't `new` a concrete class inside logic; inject the interface from outside" | High-level (brains/CEO) & low-level (details/employees) modules **both depend on an abstraction**, not each other; the "inversion" = dependency arrow points at an interface, not down into details. Analogy: order via **food app** (abstraction), not the chef directly. Example: `RecommendationEngine` that `new`s `RecentlyAdded` (coupled) → depend on `RecommendationStrategy`, inject `TrendingNow`/`GenreBased`; swap strategy at runtime, engine untouched. **DIP = the principle; Dependency Injection (DI) = the technique** (pass dependency via constructor/setter). Wins: flexibility, testability (mock the abstraction), reuse, maintainability, scalability. Pitfall: smell = `new ConcreteClass()` in business logic → move creation to edges (factory/DI/`main`); needs a **stable** abstraction; don't invert everything (YAGNI). |
