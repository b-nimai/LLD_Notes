# Single Responsibility Principle (SRP) — the **S** in SOLID

> **One-liner:** A class should have **one and only one reason to change** — one job, one responsibility.

**Trigger phrase:** "One class, one job. If you can describe a class with 'AND', split it."

> SOLID = 5 principles for clean, scalable, maintainable OOP code. SRP is the foundation the other four build on.

---

## Core Idea

A class should own **a single responsibility**. The litmus test isn't "how many methods" — it's **"how many distinct reasons would force this class to change?"** Each responsibility is a potential axis of change; bundling several into one class **couples** them, so a change to one can ripple into the others.

> **Reason to change = a stakeholder/concern.** Business rules change for one reason, the database schema for another, the UI for another. Keep them in separate classes.

---

## Real-life Analogy

A **chef** who also cleans, serves, and orders groceries does all of them poorly — busy cleaning, the food suffers. Split the roles: **chef cooks, cleaner cleans, waiter serves, manager orders**. Each focuses on one thing → better results, and you can replace the cleaner without disrupting the kitchen.

---

## Worked Example — TUF+ Compiler

A `TUFplusCompiler` that does **everything** violates SRP:
- adds driver code · runs syntax check · runs test cases · stores output in DB · returns output to user

Split by responsibility — **each class changes for exactly one reason:**

| Class | Single responsibility | Changes when… |
|-------|----------------------|---------------|
| `DriverCodeGenerator` | add driver code | driver-code format changes |
| `SyntaxChecker` | syntax checking | language grammar changes |
| `TestRunner` | run code vs test cases | execution/sandbox changes |
| `DatabaseManager` | persist output | DB/schema changes |
| `UserOutputHandler` | return output to user | response format changes |
| `Coordinator` | orchestrate the above | the *workflow* changes |

Now each piece is modular, independently testable, and replaceable without touching the rest.

---

## Why it matters (Advantages)

- **Maintainability** — a change is localized; it won't ripple across unrelated code.
- **Readability** — small, focused classes are easy to grasp.
- **Reusability** — a single-purpose class reuses cleanly, with no dependency baggage.
- **Testability** — fewer responsibilities ⇒ fewer dependencies ⇒ simpler tests.
- **Lower change-risk** — one concern per class means fewer unintended side effects.

---

## Common Violations (smell list)

- **DB logic mixed with business logic** — SQL/JDBC alongside core rules → can't swap the data layer without risking the rules.
- **UI coupled to business logic** — app logic embedded in the UI → can't restyle the screen without touching behavior.
- A class whose description needs **"AND"** ("handles orders **and** sends emails **and** logs").

---

## Pitfalls / Things to Remember

- **SRP is not "one method per class."** It's one *reason to change* — a class can have many methods serving one responsibility.
- **Don't over-split.** Shredding everything into trivially tiny classes hurts readability too — balance cohesion against fragmentation.
- **SRP scales beyond classes.** Same mindset applies to **methods, modules, microservices, whole systems** — every component should have one responsibility.
- High cohesion + low coupling is the real goal SRP serves.
