# Liskov Substitution Principle (LSP) — the **L** in SOLID

> **One-liner:** If `S` is a subtype of `T`, you must be able to **replace `T` with `S` anywhere** without breaking the program's correctness.

**Trigger phrase:** "A subclass must be a drop-in for its parent. If swapping the child in surprises the caller (wrong result, exception, do-nothing), it's an LSP violation."

---

## Core Idea

Inheritance promises substitutability: code written against a base class should keep working when handed any subclass. A subclass must **honor the parent's contract** — same expectations, no nasty surprises. If a child **changes behavior in a way that breaks the parent's assumptions**, LSP is violated even though it compiles fine.

> LSP failures show up at **runtime**, not compile time — that's what makes them dangerous.

---

## Real-life Analogy — the Pet Hotel

Policy: *"every pet can be fed, walked, and groomed."* Dogs, cats, rabbits → fine.
- Bring a **snake** 🐍 → can't walk it, grooming makes no sense, it needs live mice, not pet food. The whole process **breaks** → LSP violation (the snake doesn't satisfy the "pet" contract the system assumed).
- Bring a **hamster** → still eats, needs care, minor tweak (wheel instead of a walk). Fits the contract → **valid substitution**.

The system must *trust* that any "pet" behaves as expected; a subtype that rewrites the rules makes it fragile.

## Classic Violation — Rectangle / Square

```java
class Rectangle {
    int width, height;
    void setWidth(int w)  { width = w; }
    void setHeight(int h) { height = h; }
    int getArea() { return width * height; }
}

class Square extends Rectangle {           // "a square IS-A rectangle"… mathematically yes
    @Override void setWidth(int w)  { width = w; height = w; }  // forces both equal
    @Override void setHeight(int h) { height = h; width = h; }
}

static void printArea(Rectangle r) {
    r.setWidth(5);
    r.setHeight(10);
    System.out.println(r.getArea());       // Rectangle → 50 ✓   Square → 100 ✗
}
```
`printArea` assumes width and height are **independent** (a Rectangle invariant). `Square` breaks that invariant, so substituting it yields **100 instead of 50**. The "is-a" felt right in English but **violates the behavioral contract** → don't model it with inheritance.

## LSP Done Right — Notification System

```java
class Notification {
    public void sendNotification() { System.out.println("Notification sent"); }
}
class EmailNotification extends Notification {
    @Override public void sendNotification() { System.out.println("Email Notification sent"); }
}
class TextNotification extends Notification {
    @Override public void sendNotification() { System.out.println("Text Notification sent"); }
}

// Caller is unchanged — any subtype drops in cleanly:
Notification notification = new EmailNotification();
notification.sendNotification();
```
Each subclass **strengthens nothing / breaks nothing** — it just provides its own valid way to "send a notification." Adding a new channel = add a subclass; client code stays put. This is LSP enabling safe extension (and it's what makes [[2. Open Closed Principle]] work).

---

## Why LSP Matters

A violation makes code **unpredictable** (base-class assumptions break for some subtypes), **hard to maintain** (every new subclass forces re-checking all usages), **bug-prone** (runtime errors / wrong output), **less reusable**, and **tightly coupled** (clients start type-checking specific subclasses).

---

## How to Spot Violations (ask these)

- Does the subclass **override a method in a way that changes its meaning/assumptions**?
- Can I replace base with subclass **everywhere** without changing expected behavior?
- Does the subclass **throw unexpected exceptions** or **return wrong values**?
- Does it **weaken a precondition** or **strengthen a postcondition**? (Design-by-contract rule.)
- Tell-tale smell: an override that is **empty, throws `UnsupportedOperationException`, or returns `null`/a sentinel.**

Any "yes" ⇒ likely LSP violation.

---

## Key Rules to Follow

- Subclasses must **honor the parent's contract**.
- Don't override to **drastically change behavior**.
- A subclass should **only extend, never restrict** behavior.
- Think in terms of **interfaces and behavioral compatibility**, not just structural "is-a".
- **Prefer composition over inheritance** when the behavioral fit isn't clean.

---

## Pitfalls / Things to Remember

- **"is-a" in English ≠ valid subtype.** Square *is a* rectangle in math but not in behavior — model by *contract*, not by noun.
- Violations are **invisible at compile time**; they bite at runtime.
- The fix is often **restructuring the hierarchy** (e.g. split into separate types / interfaces) rather than patching the override.
