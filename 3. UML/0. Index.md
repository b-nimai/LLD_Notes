# UML — Index

30-second whole-topic recall. Read the **Trigger** and try to recall the **30-sec revision** before opening the note.

**UML** = standard *visual notation* for software design. Two families: **Structural (static — what exists)** & **Behavioral (dynamic — what happens)**. For **LLD, only the Class Diagram matters**.

|---|-------|----------------|-----------------|
| 1 | [1. Introduction](1.%20Introduction.md) | "UML = blueprints for software (house plan before bricks); two families — **Structural = static/what exists**, **Behavioral = dynamic/what happens**" | **UML (Unified Modeling Language)** = standardized graphic notation to **visualize, specify, construct, document** a system's structure & behavior — language-agnostic, a *diagram standard not code*. Analogy: house **blueprints** before laying bricks → design clearly, catch flaws on paper. **Two families:** **Structural = static** (what the system *contains*; photo) → Class, Object, Component, Composite Structure, Deployment, Package, Profile (7). **Behavioral = dynamic** (what *happens* over time; video) → Use Case, Activity, Sequence, Communication, State Machine, Interaction Overview, Timing (7). **Why:** shared language, catch design flaws early, onboarding. Pitfall: don't memorize all 14 — for **LLD the Class Diagram is the one that matters**; Structural vs Behavioral is the key split (static=structure, dynamic=behavior). |
| 2 | [2. UML Class Diagram](2.%20UML%20Class%20Diagram.md) | "Class = 3-box rectangle (name/attributes/methods) + visibility `+ - # ~`; the design lives in the **6 relationship arrows**" | **Class Diagram** = the static map of classes/interfaces/enums + attributes/methods + relationships — *the* LLD diagram. **Class** = 3 compartments (name / attributes / methods). **Visibility:** `+`public · `-`private · `#`protected(class+subclasses) · `~`package → encapsulation. **Syntax:** attr `vis name: Type [mult] = default` (`+ age: int = 21`); method `vis name(p: Type): Ret`. **Stereotypes:** `<<interface>>` (methods only), `<<abstract>>` (italic name), `<<enumeration>>` (literals). **3 perspectives:** Conceptual (concepts, no attrs — BAs) · Specification (interfaces + key methods — architects) · Implementation (everything — devs). **⭐ 6 relationships:** Association (USE-A, solid line) · Aggregation (HAS-A, **hollow** ◇ at whole, parts survive) · Composition (strong HAS-A, **filled** ◆, parts die with whole) · Inheritance (IS-A, **solid** line + hollow triangle→parent) · Realization (implements, **dashed** + hollow triangle→interface) · Dependency (temporary use, **dashed** + open arrow). Pitfalls: hollow vs filled diamond = lifecycle; solid vs dashed triangle-line = extends vs implements; triangle vs open-arrow = is-a vs uses. |

### Relationship cheat-sheet (the summary table)

| Relationship | Notation | Example |
|--------------|----------|---------|
| Association | solid line `——` | Student —— Teacher |
| Aggregation | hollow diamond `◇——` | Department ◇—— Professor |
| Composition | filled diamond `◆——` | House ◆—— Room |
| Inheritance | solid + hollow triangle `——▷` | Dog ——▷ Animal |
| Realization | dashed + hollow triangle `┄┄▷` | Circle ┄┄▷ Shape interface |
| Dependency | dashed + open arrow `┄┄→` | Order ┄┄→ PaymentService |
