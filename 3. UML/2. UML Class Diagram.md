# UML Class Diagram

> **One-liner:** The Class Diagram is the **one UML diagram that matters for LLD** — a static map of a system's **classes, interfaces, enums, their attributes & methods, and the relationships between them**.

**Trigger phrase:** "A class = a **3-box rectangle** (name / attributes / methods) with **visibility markers** (`+ - # ~`); the design is really in the **6 relationship arrows** — Association, Aggregation, Composition, Inheritance, Realization, Dependency."

> **Why it's *the* LLD diagram:** it shows, at a glance, what classes exist, what each can do, and how they connect — exactly the artifact an LLD interview wants you to draw.

---

## Core Idea

A **Class Diagram** is a **static, structural** view of the system: its classes/interfaces/enums, their attributes and operations, and how they relate. It's used for **domain modeling, data modeling, and architecture**, drawn early in the lifecycle and refined as the project grows. Done right, anyone can read the system's structure **without reading the code**.

---

## 1. Class representation — the 3 compartments

A class is a rectangle split into three compartments:

```
┌─────────────────────────┐
│        ClassName         │   ← name (bold, centered)
├─────────────────────────┤
│  - field: Type = default │   ← attributes
├─────────────────────────┤
│  + method(p: Type): Ret  │   ← operations (methods)
└─────────────────────────┘
```

## 2. Visibility markers

| Marker | Visibility | Reachable from |
|:------:|-----------|----------------|
| `+` | **Public** | any other class |
| `-` | **Private** | only within the class itself |
| `#` | **Protected** | the class **and its subclasses** |
| `~` | **Package** | classes in the same package |

> These markers enforce **encapsulation** — they're how a diagram shows access control.

## 3. Attribute & method syntax

**Attribute:** `visibility name: Type [multiplicity] = DefaultValue`
- `public int age = 21;` → `+ age: int = 21`
- *multiplicity* (optional): `0..1`, `1..*` — how many instances can exist.

**Method:** `visibility name(param: Type, ...): ReturnType`
- `private boolean isAdult(int age)` → `- isAdult(age: int): boolean`

Optional extras: **multiplicity, default values, stereotypes** like `<<constructor>>`, `<<static>>`.

## 4. Special types (shown via *stereotypes*)

| Type | How it's drawn | Note |
|------|----------------|------|
| **Interface** | `<<interface>>` over the name; methods only | a contract; no attributes — *except* declared constants |
| **Abstract class** | `<<abstract>>` stereotype + **italic** class name | can't be instantiated; mixes abstract + concrete methods |
| **Enumeration** | `<<enumeration>>` over the name; literals listed in a second compartment | a fixed set of named values (literals) |

## Three Perspectives (how much detail to show)

| Perspective | Audience | Shows |
|-------------|----------|-------|
| **Conceptual** | Business Analysts, Domain Experts | real-world concepts (Customer, Order, Invoice); **no attributes/methods**, business-level associations |
| **Specification** | Architects, Designers | abstract classes, interfaces, **key public methods**; contract-based design ("what a class promises") |
| **Implementation** | Developers | **everything** — all attributes, visibility markers, types, defaults, constructors, and **all** relationships |

> Memory hook: Conceptual = **what concepts**, Specification = **what contracts**, Implementation = **what code**.

---

## ⭐ The 6 Relationships (the heart of the diagram)

| Relationship | Meaning | UML Notation | Example |
|--------------|---------|--------------|---------|
| **Association** (USE-A) | one class uses/interacts with another (1-1, 1-many, many-many) | solid line | Student —— Teacher |
| **Aggregation** (HAS-A) | whole-part, but parts **survive** the whole | **hollow** diamond at the *whole* | Department ◇—— Professor |
| **Composition** (strong HAS-A) | whole-part, parts **die** with the whole | **filled** diamond at the *whole* | House ◆—— Room |
| **Inheritance** (IS-A) | subclass extends a superclass | solid line + **hollow triangle** → parent | Dog ——▷ Animal |
| **Realization** (implements) | a class fulfills an interface's contract | **dashed** line + **hollow triangle** → interface | Circle ┄┄▷ Shape |
| **Dependency** (uses temporarily) | one class uses another briefly; changes there may affect it | **dashed** line + **open arrow** → used class | Order ┄┄→ PaymentService |

> **Load-bearing distinctions** (interview favorites):
> - **Aggregation vs Composition** = diamond *color*: hollow ◇ = independent lifecycle (professors outlive the department); filled ◆ = shared lifecycle (rooms die with the house).
> - **Inheritance vs Realization** = line *style*: solid line = extends a class; **dashed** line = implements an interface. Both use the **hollow triangle**.
> - **Association vs Dependency** = **duration**: association is a *structural, lasting* link (often a field); dependency is *temporary* (a method parameter/local), shown **dashed with an open arrow**.

---

## Pitfalls / Things to Remember

- **Don't confuse the two diamonds.** Hollow ◇ = Aggregation (parts live on); Filled ◆ = Composition (parts destroyed with the whole). The *fill* is the whole answer.
- **Solid vs dashed triangle line.** Inheritance (extends) = **solid**; Realization (implements interface) = **dashed**. Same arrowhead (hollow triangle), different line.
- **Triangle vs open arrow.** Hollow **triangle** = is-a / implements; open **arrow** (`→`) = uses / depends-on. Don't mix the heads.
- **Match the perspective to the audience** — drawing every private field in a *conceptual* diagram (or stopping at concept names when an interviewer wants implementation) signals you've missed the diagram's purpose.
- **Interfaces normally have no attribute box** — the exception is when they declare **constants**.
