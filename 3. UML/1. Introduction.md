# Introduction to UML

> **One-liner:** UML (Unified Modeling Language) is the standard *visual notation* for designing software — a toolkit of diagrams that captures a system's **structure** (what it's made of) and **behavior** (how it acts), so you plan before/alongside coding.

**Trigger phrase:** "UML = blueprints for software — like a house plan shows rooms/pipes/wiring before bricks, UML diagrams show classes/objects/interactions before code. Two families: **Structural = static (what exists)**, **Behavioral = dynamic (what happens)**."

> **Setup for this track:** UML is **language-agnostic** — it's a *diagramming standard*, not code. For LLD, only one diagram really matters: the **Class Diagram** (covered next). The rest are good to *recognize*, not memorize.

---

## Core Idea

**UML** is a standardized modeling language to **visualize, specify, construct, and document** the structure and behavior of software systems. It gives you a fixed set of **graphic notations** so any developer reads the same diagram the same way — covering both **static** (structure) and **dynamic** (behavior) aspects.

Think of it as a **map before a journey**: you see where everything is and how it connects before you start moving.

### Real-life analogy
Building a house → you draw **blueprints** first (where rooms go, how pipes connect, how electricity flows). Skip them and things get messy, expensive, and confusing. **UML diagrams are the blueprints of software** — design clearly, avoid confusion, and catch design problems *before* a line of code is written.

---

## The Two Families of UML Diagrams

UML diagrams split into two top-level categories:

| Family | Captures | Analogy | Focus |
|--------|----------|---------|-------|
| **Structural** | the **static** structure — what the system *contains* and how parts relate | architectural blueprint (foundation, components, connections) | **things that exist** — classes, objects, hardware |
| **Behavioral** | the **dynamic** behavior — how it acts *over time*, how users interact, how parts communicate | a movie's script + animation (what happens, when, who) | **actions** — interactions, processes, state changes |

> **Memory hook:** Structural = a **photo** (frozen snapshot of what's there). Behavioral = a **video** (what unfolds over time).

---

## Structural Diagrams (7) — *what exists*

| Diagram | One-line purpose |
|---------|------------------|
| **Class Diagram** ⭐ | classes, their attributes, methods & relationships — the map of the code structure |
| **Object Diagram** | a *snapshot* of concrete instances and their links at one point in time |
| **Component Diagram** | how software components/modules are organized & connected |
| **Composite Structure** | the internal parts of a class and how they interact to do its job |
| **Deployment Diagram** | how software is physically deployed onto hardware/servers |
| **Package Diagram** | groups related elements (e.g. classes) into packages for organization |
| **Profile Diagram** | extends/customizes UML for a specific platform or domain |

## Behavioral Diagrams (7) — *what happens*

| Diagram | One-line purpose |
|---------|------------------|
| **Use Case Diagram** | what users (actors) can do — high-level functionalities |
| **Activity Diagram** | workflows / business processes — like a flowchart |
| **Sequence Diagram** | the *order* of messages exchanged between objects over time |
| **Communication Diagram** | interactions between objects, emphasizing how they're connected |
| **State Machine Diagram** | how an object transitions between states on events |
| **Interaction Overview** | mixes sequence + activity to model interaction flow |
| **Timing Diagram** | object behavior with respect to time (great for real-time systems) |

---

## Why it matters

- **Shared language** — a class diagram reads the same to every engineer, regardless of who wrote the code; great for onboarding & stakeholder buy-in.
- **Catch flaws on paper** — fixing a design mistake in a diagram is far cheaper than after it's coded.
- **The LLD lens** — interviews and detailed design live almost entirely in the **Class Diagram**.

---

## Pitfalls / Things to Remember

- **UML ≠ code.** It's a *notation/standard* for communicating design — language-agnostic, no execution.
- **Don't memorize all 14 diagrams.** For **LLD, the Class Diagram is the one that matters** — the rest you only need to *recognize*. (The course note itself flags this.)
- **Structural vs Behavioral is the key split** — if you can't immediately say which family a diagram belongs to, you don't yet know what it's *for*. Static = structure, dynamic = behavior.
- **It's a design aid, not the design.** A pretty diagram with a bad class breakdown is still a bad design — UML only *shows* the thinking, it doesn't do it for you.
