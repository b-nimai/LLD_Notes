# Software Design Principles — DRY · KISS · YAGNI

> **One-liner:** Three universal guidelines for code that's easy to understand, maintain, and extend — **don't duplicate (DRY)**, **don't overcomplicate (KISS)**, **don't build what you don't need yet (YAGNI)**.

**Trigger phrase:** "DRY = one source of truth (no copy-paste logic) · KISS = simplest thing that works (no clever code) · YAGNI = build it only when actually needed (no speculative features)."

> These are **mindset** principles (apply at both HLD and LLD), broader than SOLID. Each one also has a **"when NOT to"** — they're heuristics, not laws.

---

## 1. DRY — Don't Repeat Yourself

**Every piece of knowledge must have a single, unambiguous, authoritative representation in the system.** In short: don't duplicate logic. Duplicated code means that when logic changes you must hunt down *every* copy — miss one and you have a bug.

**Why it matters:** reduces redundancy · easier maintenance · **single point of change**.

### Before (violates DRY) → After (DRY)
```java
// BAD — area logic written out twice
int area1 = length1 * width1;
System.out.println("Area1: " + area1);
int area2 = length2 * width2;
System.out.println("Area2: " + area2);   // change the formula → fix it in 2 places
```
```java
// GOOD — one authoritative definition
class AreaCalculator {
    public static int calculateArea(int length, int width) {
        return length * width;            // change it here, everywhere updates
    }
}
int area1 = AreaCalculator.calculateArea(10, 5);
int area2 = AreaCalculator.calculateArea(8, 4);
```

### Applying DRY in practice
- Spot repeated code → replace with **one reusable method/class**.
- Extract common functionality into **methods or utility classes**.
- Lean on **libraries/frameworks** instead of re-implementing.
- Refactor duplicate logic **regularly** across classes and layers.

### When NOT to apply DRY
- **Premature abstraction** — two blocks that *look* alike today may diverge tomorrow. Forcing them into one method creates **coupling between unrelated code**. (Rule of thumb: prefer the *rule of three* — wait for the third repetition.)
- **Performance-critical code** — a generalized wrapper adds call overhead/indirection and can block compiler **inlining**; repeating tuned low-level logic may be faster.
- **Readability cost** — if the shared abstraction is harder to follow than the duplication, **prefer clarity**.
- **Legacy code** — extracting shared logic without tests can silently change behavior; *"leave it alone unless you must touch it."*

> **Key nuance:** DRY is about a single source of *knowledge*, not just identical-looking *text*. Two pieces of code that coincidentally match aren't really a duplication.

## 2. KISS — Keep It Simple, Stupid

**Simplicity is a design goal; avoid unnecessary complexity.** Use the simplest solution that works — no clever, convoluted code that the next person (or future you) has to decode.

**Why it matters:** easier debugging · improved readability · better maintainability · faster development.

### Before (overengineered) → After (simple)
```java
// BAD — extra variable + redundant if/else for a boolean
public static boolean isEven(int number) {
    boolean isEven = false;
    if (number % 2 == 0) { isEven = true; } else { isEven = false; }
    return isEven;
}
```
```java
// GOOD — the expression already IS the answer
public static boolean isEven(int number) {
    return number % 2 == 0;
}
```

### How to keep it simple
- Let an expression be the answer instead of wrapping it in control flow.
- Prefer clear names and small functions over "smart" one-liners that hide intent.
- Don't reach for a design pattern/framework when a plain method does the job.

> **KISS ≠ simplistic.** The goal is the *simplest solution that fully solves the problem* — not cutting corners. Solve the real requirement, just without the extra machinery.

## 3. YAGNI — You Aren't Gonna Need It

**"Always implement things when you actually need them, never when you just foresee that you'll need them."** Don't build features on speculation — it bloats the codebase and wastes time on things that may never ship (or ship differently).

**Why it matters:** reduced waste · simpler codebase · faster development.

### Example
You're asked to build a note app that can **create** and **view** notes. Then the "what if" spiral starts: *"What about categories? Tagging? Google Drive sync? Let me prepare for all of it!"* — none of it was asked for, and you've now added complexity and delay for features that may never arrive. **Build create + view. Add the rest when it's actually requested.**

### When NOT to apply YAGNI
- **Well-known, committed requirements** — if a feature is *guaranteed and imminent*, preparing now can save a painful refactor. *Example:* messaging supports text today, but the team has **committed to image support in 2 sprints** → designing the data model for attachments now is reasonable.
- **Performance-critical areas** — where performance is a first-class concern, building and testing realistic usage patterns early can surface bottlenecks before they're expensive to fix.

> **YAGNI vs good design:** YAGNI targets *speculative features*, not *clean structure*. Keep your design extensible; just don't pre-build concrete features nobody asked for yet.

## How the three fit together

| Principle | Fights | Core question | One-line test |
|-----------|--------|---------------|---------------|
| **DRY** | Duplication | "Is this knowledge defined once?" | Would a change force edits in 2+ places? |
| **KISS** | Complexity | "Is this the simplest thing that works?" | Can a teammate read it without decoding? |
| **YAGNI** | Speculation | "Do I need this *now*?" | Was this feature actually requested? |

They can mildly tension each other (over-DRYing hurts KISS; KISS may tolerate tiny duplication). When in doubt: **clarity wins**, and you apply judgment — these are guidelines, not absolutes.

---

## Pitfalls / Things to Remember

- **Don't cargo-cult DRY** — premature abstraction couples unrelated code; the *rule of three* is safer than deduplicating on sight.
- **KISS is about the solution, not the problem** — simple code must still fully solve the requirement.
- **YAGNI ≠ "no planning"** — keep architecture extensible; just don't *implement* unrequested features.
- Every principle here has a **legitimate "when not to"** — applying them blindly is itself a design smell.
