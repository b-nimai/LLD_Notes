# Introduction to Low-Level Design (LLD)

> **One-liner:** LLD is the detailed, code-level blueprint — classes, methods, data structures — that turns the high-level architecture into something you can actually implement.

**Trigger phrase:** "HLD = architect's blueprint (where rooms go); LLD = wiring/plumbing (exact switches, materials) — the small detailed planning *before* you write code."

> **Setup for the whole track:** examples are in **Java** (most big codebases use it), but every concept is language-agnostic. The one hard prerequisite is a **solid grip on OOP** — LLD is basically OOP applied at interview altitude.

---

## Core Idea

**Low-Level Design** is the phase of the software development lifecycle that specifies *how* each module is built — its **internal structure, classes, methods, algorithms, and data structures**. It is the **bridge between High-Level Design and actual code**.

The house analogy makes the HLD↔LLD split click:

| | House | Software |
|---|---|---|
| **HLD** | Architect's blueprint — where rooms go, how big, how connected | System overview — modules, services, how they talk |
| **LLD** | Where switches go, how plumbing is laid, what materials | Classes, methods, fields, data structures, algorithms |

So **HLD answers "what are the pieces and how do they connect"**, while **LLD answers "how is each piece actually built."**

### Concrete example
A **Login System** at HLD is just a box labelled "Authentication." At LLD it becomes real components with defined behavior:
- `login(username, password)` — validate credentials, start a session
- `signUp(...)` — create a new user, enforce uniqueness/validation
- `forgotPassword(email)` — generate + send a reset token

LLD says *which classes hold these methods, what they return, and what happens on failure.*

---

## Key Characteristics of LLD

1. **Granular & code-level** — defines actual classes, functions, variables, and data structures. Not "we need authentication," but "this class handles it, this method validates login, this is what happens on failure."
2. **Implementation-focused** — directly maps to the code that gets written. Often expressed with **pseudocode, flow diagrams, and sequence diagrams** showing real data flow between functions.
3. **Applies OOP heavily** — leans on **encapsulation, abstraction, inheritance, polymorphism** to build **modular, reusable, maintainable** systems. *Example:* a base `Notification` class with `EmailNotification` and `SMSNotification` subclasses (inheritance + polymorphism).

> **Stakeholders:** the people who actually implement the system — **senior developers, tech leads, engineering managers** (not business/product, who live mostly at HLD).

---

## HLD vs LLD (at a glance)

| Aspect | High-Level Design (HLD) | Low-Level Design (LLD) |
|--------|-------------------------|------------------------|
| **Purpose** | System overview & modules | Detailed implementation & logic |
| **Level of detail** | Abstract | Highly detailed |
| **Focus** | Architecture, modules, interfaces | Class diagrams, methods, data structures |
| **Outcome** | Module/system diagrams | Detailed class & method diagrams |
| **Audience** | Architects, stakeholders | Developers, tech leads |

---

## Why LLD Matters

- **Avoids rework** — catching design flaws on paper is far cheaper than after the code is written.
- **Improves collaboration** — a shared reference so the whole team agrees on component behavior and integration points.
- **Promotes scalability** — clean, modular components absorb growth and new features without a rewrite.
- **Encourages best practices** — pushes clean code, design patterns, and OOP, giving maintainable, robust systems.
- **Career relevance** — LLD is the core of **senior-engineer interviews**; it's where OOP + SOLID + design patterns are actually tested.

---

## Pitfalls / Things to Remember

- **LLD ≠ HLD.** HLD is the *what/where* (abstract, architecture); LLD is the *how* (detailed, class-level). Mixing altitudes in an interview is a common mistake.
- **LLD is not "just coding."** It's the *plan* for the code — the thinking you do before typing, so you don't paint yourself into a corner.
- **Don't over-detail too early.** LLD should be detailed, but agonizing over every variable before the design stabilizes wastes time.
- **OOP is the foundation** — weak OOP = weak LLD. Revisit encapsulation/abstraction/inheritance/polymorphism first.
